# Контент и медиа: что нужно подготовить для наполнения

> Документ-спутник к `PLAN_SIDE_SELECTION_UI.md`. Здесь — конкретные файлы (медиа, тексты, JSON), которые надо собрать, чтобы залить проект реальными раскидками. Плюс ответ на вопрос про YouTube vs MP4-хост.

## 0. TL;DR

1. **Видео раскидок храним как MP4 (H.264, без звука, 720p, 5–15 сек)**, а не как YouTube-ссылки. YouTube под этот UX плохо подходит (см. §3).
2. Хостить рекомендую **Cloudflare R2** (или Bunny.net) с кастомным CDN-доменом — текущий код менять не нужно: тег `<video src="...">` уже умеет играть прямой MP4.
3. Помимо самих видео нужно: **постер-кадры (JPG/WebP)**, **скриншоты прицеливания**, **иконки гранат / сторон / категорий callouts**, **OG-превью**, **favicon-набор**, **gif-превью для ховера на главной** (опционально).
4. Текстовый контент: для каждой раскидки — `title`, `description`, `position_ids`, локализации `ru/en/uk`. Для каталога позиций — `label`, `label_i18n`, `aliases`, `hotspot.x/y`.
5. Папка `public/uploads/grenades` отлично работает на dev. Перед продом всё переезжает в R2/Bunny + CDN, ссылки в JSON остаются такими же `/media/...`, только меняется базовый URL.

## 3. Хостинг видео: YouTube vs остальное

### 3.1. Почему YouTube НЕ оптимален для UX «раскидки»

| Симптом | Объяснение |
| --- | --- |
| Логотип YouTube в углу, кнопки `Watch later`, `Share` | Параметр `modestbranding=1` Google в 2024 фактически отключил, лого остаётся. |
| «Похожие» в конце ролика | `rel=0` теперь показывает похожие **с того же канала**, а не убирает их совсем. |
| Реклама (преролл / мидролл) | Может появляться даже на 6-секундных роликах канала без монетизации — Google вставляет «свою». |
| Автоплей со звуком блокируется | Нужно `mute=1`, иначе iOS/Safari/Chrome просто не запустят. |
| Лаг на повтор (loop) | `loop=1&playlist=ID` даёт **~0.7–1.5 сек чёрного экрана** между циклами — заметно на 6-сек клипе. |
| Iframe тяжёлый | +400–700 мс инициализации до первого кадра, плюс ~500 КБ JS. |
| Контролы перекрывают суть | На мобильном жирная нижняя панель закрывает половину раскидки. |
| Нельзя отдать `<video poster>` | Превью кадры приходится делать самому. |
| Embeds могут быть отключены каналом / страной | Жёсткий риск «контент недоступен в вашем регионе». |

### 3.2. Если всё-таки нужен YouTube

Тогда используем **YouTube Shorts / обычные ролики** через `youtube-nocookie.com` embed и режем UI вокруг плеера, чтобы скрыть лишнее. Минимально нужный embed-URL:

```
https://www.youtube-nocookie.com/embed/VIDEO_ID
  ?autoplay=1
  &mute=1
  &controls=0
  &modestbranding=1
  &rel=0
  &playsinline=1
  &loop=1
  &playlist=VIDEO_ID   ← обязательно дублировать ID, иначе loop не работает
  &start=3&end=12      ← если в ролике несколько раскидок подряд
```

В коде это значит: новый компонент `LineupVideo`, который смотрит на `media_url`:

- если URL содержит `youtube.com/watch?v=…`, `youtu.be/…` или `youtube-nocookie.com/embed/…` → рендер `<iframe>`;
- иначе — текущий `<video>`.

### 3.3. Сравнение реальных вариантов

| Хост | Цена | Плюсы | Минусы | Рекомендация |
| --- | --- | --- | --- | --- |
| **Cloudflare R2** + кастомный домен | $0.015/ГБ хранения, **0 egress** | Дёшево даже на 1 ТБ; отдаёт MP4 напрямую → текущий код не меняем; свой домен (`media.tvoy-domen.ru`); CDN глобальный. | Нужно один раз настроить bucket + DNS. | ⭐ **Лучший выбор** для стартового MVP. |
| **Cloudflare Stream** | $5 / 1000 мин хранения + $1 / 1000 мин просмотров | Адаптивный битрейт (HLS), свой плеер `<stream>`, без watermarks, авто-постер. | Платный с первых минут; нужен свой компонент-обёртка. | Когда трафик станет большим (>10k/день). |
| **Bunny.net Stream / Storage** | Storage от $0.01/ГБ, Stream $0.01/ГБ + $1/мес минимум | Очень дёшево; готовый плеер `iframe.mediadelivery.net`; глобальный CDN; геозоны. | UI личного кабинета попроще, чем у CF. | Хорошая альтернатива CF, особенно если хочется готовый плеер с превью. |
| **Vercel Blob** (Next.js native) | 1 ГБ + 10 ГБ трафика бесплатно, дальше $0.15/ГБ | Деплой в один клик, SDK из коробки. | Дорогой egress на масштабе; нет встроенного плеера/HLS. | Норм для пилота на 50–100 видео, потом мигрировать. |
| **Self-host в `public/uploads/`** | Зависит от хостинга проекта | Уже работает; ноль внешних зависимостей. | На Vercel — 100 МБ лимит на функцию, статикой ~1 ГБ суммарно; на VPS зависит от диска и канала. | Только для dev / локальной разметки. |
| **YouTube unlisted** | Бесплатно | CDN Google, не платишь ничего. | Лого, реклама, лаг на loop, embed-блокировки (см. §3.1). | Только если бюджет = 0 и UX компромиссы ок. |
| **Streamable** | Free-тир урезали (15 мин, 250 МБ всего) | Простые embed-ссылки. | Бесплатный план нерабочий для проекта такого размера. | ❌ Не подходит. |
| **Imgur / Gfycat / Giphy** | — | — | Imgur режет качество и сжимает; Gfycat закрылся в 2023; Giphy ограничивает длину/размер. | ❌ Не подходит. |

> **Итоговая рекомендация:** хранить MP4 в **Cloudflare R2**, отдавать через кастомный домен (например `https://media.cs2grenades.app/`), а в `video_url` писать просто `/media/de_mirage/connector_smoke_v1.mp4`. Это:
> - не требует изменений в плеере,
> - даёт полный контроль над UX (loop без задержки, без лого, без рекламы),
> - стоит копейки даже на росте.

### 3.4. Если когда-нибудь захочется и YouTube, и MP4 одновременно

Расширяем тип раскидки минимально:

```ts
// В CustomThrowVariant / Grenade
media_url: string                    // текущее поле, остаётся как было
media_kind?: 'mp4' | 'youtube' | 'iframe' // опционально, авто-детект если не задано
```

Дальше в `BottomSheet` фабрика плеера:

```ts
function detectMediaKind(url: string): 'mp4' | 'youtube' | 'iframe' {
  if (/youtube\.com|youtu\.be/.test(url)) return 'youtube'
  if (/\.(mp4|webm|mov)(\?|$)/i.test(url)) return 'mp4'
  return 'iframe'
}
```

Это **запасной** план, не нужный, если выбираем R2.

## 4. Видео раскидок — продакшн-рецепт

### 4.1. Параметры одного клипа

| Параметр | Значение |
| --- | --- |
| Контейнер / кодек | `.mp4`, **H.264 (AVC) High profile**, faststart |
| Разрешение | **1280×720** (16:9) — этого достаточно, не пихаем 4K |
| FPS | 60 (если запись CS2 в 60+) или 30 (если конвертим OBS-захват) |
| Битрейт | 2–4 Мбит/с (CRF 22–24 у `ffmpeg`) |
| Аудио | **Без звука** (`-an`) — на сайте всё равно `muted` |
| Длина | **5–15 сек**: только сам бросок + полёт + landing. Без интро. |
| Старт ролика | **первый кадр = поза игрока перед броском** (он же будет постером) |
| Размер файла | целевой **1–3 МБ** |

Шаблон `ffmpeg`:

```bash
ffmpeg -i input.mkv \
  -vf "scale=1280:720:flags=lanczos" \
  -c:v libx264 -preset slow -crf 22 -profile:v high -movflags +faststart \
  -an \
  -t 12 \
  out.mp4
```

### 4.2. Соглашение об именах файлов

Рекомендую структурированный путь — потом легко чистить и мигрировать:

```
/media/<map_id>/<side>/<position_id>/<grenade_type>/<short_label>_v<N>.mp4
```

Примеры:

```
/media/de_mirage/t/apartments/smoke/window_v1.mp4
/media/de_mirage/t/apartments/smoke/window_v2.mp4    ← альтернативный вариант
/media/de_mirage/ct/connector/flash/pop_v1.mp4
/media/de_inferno/t/banana/molotov/coffins_v1.mp4
```

В JSON-данных кладём именно такие относительные ссылки — это упрощает ревью и резервное копирование.

### 4.3. Постер-кадр (preview)

К каждому видео — **JPG/WebP постер 1280×720**, ~80–150 КБ. Кладём рядом с видео, имя `*.poster.jpg`. Используется в `<video poster="...">` и для предзагрузки в списках.

```
/media/de_mirage/t/apartments/smoke/window_v1.mp4
/media/de_mirage/t/apartments/smoke/window_v1.poster.jpg
```

Извлечение постера из видео:

```bash
ffmpeg -i out.mp4 -ss 0.1 -vframes 1 -q:v 3 out.poster.jpg
```

### 4.4. Опционально: GIF-превью для ховера

На главной / в списке раскидок может быть приятный «mini-loop» 2–3 сек, 320×180. Это **не GIF**, а маленький MP4 (GIF в разы тяжелее). Имя `*.preview.mp4`, ~150 КБ. Использовать только если нужна вау-полировка — это V2.

## 5. Скриншоты «как целиться»

Многие раскидки стандартизуют не позицию ног, а **прицел + ориентир**. Под это:

- **`aim_screenshots/`** — JPG 1280×720, кропнутые скрины внутри игры с белой/жёлтой стрелкой, где должен быть прицел.
- 1 скрин на каждый `throw_variant` (в среднем).
- Кладём в `gallery_urls` — текущая галерея в `BottomSheet` уже листает картинки.

Имя:

```
/media/de_mirage/t/apartments/smoke/window_v1.aim.jpg
```

> Совет: делать скрины из CS2 с командой `sv_cheats 1; cl_drawhud 0` и потом подрисовывать стрелку в любом редакторе.

## 7. Каталог позиций (callouts) — JSON-наполнение

Это самая трудоёмкая текстовая часть. План UI требует справочник `src/data/positions.ts`. Каждая запись:

```ts
{
  id: 'apartments',
  map: 'de_mirage',
  side: 'T',
  category: 'b_site',
  label: 'Apartments',
  label_i18n: { ru: 'Апартаменты', en: 'Apartments', uk: 'Апартаменти' },
  aliases: ['apps', 'апки', 'апи', 'апартаменты'],
  hotspot: { x: 0.18, y: 0.18, radius: 0.05 },
}
```

### 7.1. Что нужно собрать на каждую позицию

| Поле | Источник | Кто заполняет |
| --- | --- | --- |
| `id` (slug) | сам, латиницей `_` | автор контента |
| `label` (англ.) | Liquipedia / Scope.gg | автор контента |
| `label_i18n.ru` | каноничный комьюнити-перевод | автор контента |
| `label_i18n.uk` | каноничный укр. перевод (часто — то же латиницей) | автор контента (или пропустить → fallback на en) |
| `aliases` | сленг + опечатки + транслит | автор контента |
| `hotspot.x/y` | мышью на радаре через будущий «debug-режим» админки | автор контента |
| `hotspot.radius` | по умолчанию `0.05`, для крупных зон — `0.07` | автор контента |
| `category` | один из `spawn / a_site / b_site / mid / rotation / utility` | автор контента |

### 7.2. Стартовый объём (по плану UI §7)

| Карта | Позиций (примерно) |
| --- | --- |
| Mirage | 22 |
| Dust2 | 22 |
| Inferno | 24 |
| Nuke | 18 (плюс `layer_id`) |
| Overpass | 18 |
| Anubis | 18 |
| Ancient | 14 |
| **Итого** | **≈ 136 позиций** |

### 7.3. Хелпер для разметки

В админке планируется кнопка «положить новый hotspot мышью» — пока не реализована, временно можно делать так:

1. Открыть карту в браузере с DevTools.
2. На радаре навести курсор → во встроенном debug-логе уже выводятся нормализованные координаты (см. `useRadarImageBox`).
3. Скопировать `x/y` в JSON.

## 12. Ответ на исходный вопрос (кратко)

> «Привязать к YouTube или взять другой хост?»

- **Если задача — наполнить быстро и бесплатно**, *и* ты готов мириться с лого + потенциальной рекламой + 1-сек паузой между повторами — ок, сделать `LineupVideo` с auto-detect MP4 vs YouTube embed (см. §3.4) и заливать unlisted-ролики на канал.
- **Если задача — продуктовый UX как в TikTok / Faceit Lineups** (мгновенный loop, без лого, без рекламы, превью + постер) — нужен MP4-хост. Самый дешёвый и удобный сейчас — **Cloudflare R2 + кастомный домен**. Текущий код проекта менять не придётся вообще: `media_url` будет указывать на `https://media.<домен>/...mp4`.
- Я **рекомендую второй вариант**. Он же укладывается в инфраструктуру, которую ты уже построил (`/api/admin/upload`, `<video>`-плеер, поля типа в `Grenade`).
